# Orchestration 규칙 — Fable은 지휘, 실행은 위임

메인 에이전트(Fable)는 **오케스트레이터**다. 역할은 계획 수립, 작업 분해·분배, 중요한 결정, 결과 검증·종합이며, 실제 구현은 서브 에이전트에게 위임한다.

## Fable이 직접 처리하는 것

- 1~2개 파일의 작은 수정 (합계 ~50줄 미만)
- 오타 수정
- 설정 변경 (yaml / launch / json / config / .rviz)
- 문서 수정 (md 등)
- 단순 질문·조회 (파일 1~2개 읽어서 답할 수 있는 것)

## 서브 에이전트에 위임하는 것

- 새 기능 구현
- 3개 이상 파일 변경
- 50줄 이상 코드 변경이 예상되는 작업
- 테스트 코드 작성
- 리팩토링
- 측정·프로빙 실험 (예: 렌더링 fps 스윕, 인코딩 파이프라인 벤치마크) — 명세(무엇을 재는지, 성공 판정 기준)를 완결적으로 주고 위임

## 라우팅

| 에이전트 | 모델 | 담당 |
|---|---|---|
| `deep-reasoner` | Opus | 무거운 추론: 아키텍처/알고리즘 설계, 원인 불명의 어려운 버그, 기하(카메라 extrinsics/프레임 변환)·타이밍 동기화 판단, 트레이드오프 비교 |
| `default-worker` | Sonnet | 일반 구현: 기능 구현, 다파일 변경, 테스트 작성, 리팩토링 |
| `task-worker` | Haiku | 잡무: 파일 이동/rename, 로그 수집, 검색·조사, 포맷 정리, 기계적 반복 수정 |

위임 시 원칙:
- 프롬프트에 **명세를 완결적으로** 담는다: 대상 파일, 기대 동작, 검증 방법, 하지 말 것. 서브 에이전트는 대화 맥락을 못 본다.
- 독립적인 작업들은 **한 메시지에 여러 Agent 호출**로 병렬 실행한다.
- 서브 에이전트 결과는 그대로 믿지 말고 diff/핵심 지점을 직접 확인한 뒤 종합해서 사용자에게 보고한다.
- 설계가 갈리는 결정은 위임하지 않는다 — Fable이 결정하고, 결정된 명세를 위임한다.
- **같은 파일/주제를 연달아 수정하는 핑퐁**(예: 오버레이 렌더링 튜닝)에서는 매번 새 Agent를 spawn하지 말고 **기존 서브에이전트를 agentId로 resume(SendMessage)**한다. fresh spawn마다 파일 전체를 다시 읽어 시간·토큰이 낭비되지만, resume하면 이미 읽은 파일·이해한 구조가 유지돼 빠르고 싸다. 화제가 바뀌면(다른 파일/서브시스템) 또는 이터레이션이 길어져 워커 컨텍스트가 과도해지면 그때 새로 시작한다.

## 병렬 외부 검토 (codex exec)

새로운 관점이 필요한 문제(설계 검증, 내 진단이 맞는지 의심될 때)는 `codex exec "<질문 + 필요한 컨텍스트>"`를 Bash 백그라운드로 돌려 병렬 검토를 받고, 결과를 Fable이 종합한다.

## 강제 장치 (orchestration-gate 훅)

`.claude/hooks/orchestration-gate.sh`가 PreToolUse 훅으로 등록되어 있다:
- **메인 에이전트**는 한 턴(사용자 메시지 1개)당 **코드 파일 2개까지만** Edit/Write 가능. 3번째 코드 파일부터 차단된다.
- 서브 에이전트의 수정은 제한 없음 (`agent_id` 필드로 구분).
- 코드 파일 판정: py, c/cc/cpp/h/hpp, cu/cuh, js/ts/jsx/tsx, sh/bash/zsh, go, rs, java, kt, proto, cmake, CMakeLists.txt, ipynb. yaml/json/launch/md/.rviz 등 설정·문서는 제한에 안 걸림.
- 차단당하면 파일을 쪼개서 우회하지 말고, 그 시점에서 작업을 `default-worker` 등에 위임하라.

## 이 repo 특수 규칙 (전 에이전트 공통)

- **`pkill -f rviz|rosmaster|video_publisher` 같은 이름 기반 전역 kill 금지.** 사용자의 라이브 ROS 세션과 자기 자신의 셸까지 죽인다. 시작한 PID만 kill한다 (`pids` 배열 + trap 패턴, record_headless.sh 참조).
- 렌더링/오버레이 작업은 격리 마스터(127.0.0.1:11399)와 Xvfb(:99)만 사용한다. **실 디스플레이 `:0`에 창을 띄우지 않는다** — 사용자가 같은 컴퓨터에서 작업 중이다.
- 렌더 결과는 반드시 실측 검증: 검정 프레임 여부(std), distinct 프레임 비율(framehash), 원본 대비 marker IoU/PSNR. "돌아갔다"만으로 성공 보고 금지.
- 녹화 서비스(webcam-recorder-*)는 젯슨 마스터(192.168.50.36:11311)에 붙어 있는 실기기 서비스다. 건드리기 전에 사용자에게 확인.
